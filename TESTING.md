with specifics on the Playwright browser suite and interactive CLI debugging.

## Port Discovery

When running `just dev-up`, Vite may select a different port if the default (3000/3001)
is already in use. The startup output shows the actual port:

```
╔═══════════════════════════════════════════════════════════╗
║  Services ready:                                        ║
║    Worker:  http://localhost:8787                      ║
║    Site:    http://127.0.0.1:3002                      ║
╠═══════════════════════════════════════════════════════════╣
║  playwright-cli:                                        ║
║    playwright-cli open http://127.0.0.1:3002            ║
║  Tip: Use state-save/state-load to persist sessions    ║
╚═══════════════════════════════════════════════════════════╝
```

Always use the port shown in this banner when running `playwright-cli open` or
navigating to auth-gated pages.

## UI Tests (`tests/ui/`)

Playwright browser tests for the React portal and admin console.
The site is started automatically, but **the API worker is not** — start it
with `just dev-up` in a separate terminal first.

```bash
just dev-up        # terminal 1
just test-ui       # terminal 2
```

### Test structure

- `_audit-helpers.ts` — shared mocks (`mockSession`, `mockPortal`, `mockAdmin`)
- `portal-*.test.ts` — portal (customer) UI
- `admin-*.test.ts` — admin console UI
- `dashboard.test.ts` — cross-cutting navigation
- `ai-guidance*.test.ts` — AI guidance surfaces

All portal tests call `mockPortal(page)` before `page.goto()`, which stubs
`/auth/me`, `/api/v1/tenant`, and every API endpoint the portal touches. This
lets the SPA render the authenticated layout immediately without a real backend.

## Playwright CLI — Auth-Gated SPA

The portal is a React SPA with client-side auth guards. When unauthenticated,
the router redirects to `/login` **before any API request is fired**. This
matters for interactive `playwright-cli` sessions because the `route` command
only intercepts **network requests** — it cannot intercept a `useEffect` that
calls `navigate('/login')`.

### What does NOT work

```bash
# This fails — the SPA redirects to /login before any API call:
playwright-cli route "**/api/v1/**" --body='{...}'
playwright-cli goto "http://127.0.0.1:3000/portal/agents/..."
# Result: /login page, not the portal
```

### Three approaches that DO work

#### 1. Cookie injection (fastest for one-offs)

The local dev stack seeds a tenant user. Log in via the API, grab the
`fp_session` cookie, then inject it into the browser before navigating to
auth-gated pages:

```bash
# 1. Read the tenant user password from .dev.vars
PASS=$(grep 'O11YFLEET_SEED_TENANT_USER_PASSWORD' apps/worker/.dev.vars | cut -d= -f2)

# 2. Authenticate and extract the Set-Cookie header
COOKIE=$(curl -s -X POST http://localhost:8787/auth/login \
  -H "Content-Type: application/json" \
  -H "Origin: http://localhost:8787" \
  -d "{\"email\":\"demo@o11yfleet.com\",\"password\":\"$PASS\"}" \
  -D - | grep -i "set-cookie" | sed 's/Set-Cookie: //;s/;.*//')

# 3. Open browser, inject cookie, then navigate
playwright-cli open http://127.0.0.1:3000
playwright-cli cookie-set fp_session "${COOKIE#fp_session=}" \
  --domain=127.0.0.1 --path=/ --httpOnly

# 4. Now the SPA sees the session and renders the portal
playwright-cli goto "http://127.0.0.1:3000/portal/agents/{configId}/{agentUid}"
playwright-cli snapshot
```

> The password is generated by `just ensure-dev-secrets` on first `just dev-up`.
> Always read it from `apps/worker/.dev.vars` — never hardcode.

#### 2. Save/load an authenticated session (best for repeated work)

Log in manually once, save the browser state, then reuse it:

```bash
playwright-cli open http://127.0.0.1:3000/login
# ... complete email or GitHub login in the opened browser ...
playwright-cli state-save auth-session.json
playwright-cli close

# Future sessions
playwright-cli open http://127.0.0.1:3000 --config=auth-session.json
playwright-cli goto "http://127.0.0.1:3000/portal/overview"
```

#### 3. Mock at test-code level (what the automated suite uses)

In test code, mocks are registered **before** `page.goto()`:

```typescript
// In _audit-helpers.ts — called before every portal test
await mockJson(page, "/auth/me", {
  user: { userId: "u-demo", role: "member", tenantId: "t-demo" },
});
await page.goto("/portal/agents/..."); // SPA renders authenticated layout
```

The Playwright CLI equivalent (using `eval` to manipulate `localStorage`)
is possible but clumsy. Prefer cookie injection (approach 1) for CLI use.

### Why route-only mocking works in test code but not CLI

In test code we call `mockSession(page)` **before** `page.goto()`. The mock is
registered on the `Page` instance, so when the SPA eventually makes its first
`/auth/me` fetch (after the initial render + auth-guard redirect), the response
is already intercepted. With the CLI, `route` commands run **after** the page
has already loaded and redirected.

### After auth: mocking API responses

Once the cookie is in place, `route` works normally because the SPA is already
inside the auth gate:

```bash
# Mock the agent detail endpoint
playwright-cli route "**/api/v1/configurations/{configId}/agents/**" \
  --body='{"available_components":{"components":{"receivers":{"sub_component_map":{"otlp":{}}}}}}' \
  --status=200

# Reload to hit the mocked endpoint
playwright-cli reload
playwright-cli snapshot
```

### CLI reference

| Command                                   | When to use                                              |
| ----------------------------------------- | -------------------------------------------------------- |
| `cookie-set`                              | Injecting session cookies (required for auth-gated SPAs) |
| `state-save` / `state-load`               | Reusing a real login session across sessions             |
| `route`                                   | Mocking API responses **after** auth is established      |
| `eval "window.localStorage.setItem(...)"` | Last resort — injecting SPA state directly               |

## Debugging a failing UI test

```bash
# Run a single test file with headed browser (no headless)
cd tests/ui && npx playwright test src/portal-interactions.test.ts --headed

# Step through with the inspector
cd tests/ui && npx playwright test src/portal-interactions.test.ts --ui

# Show the trace from a failed run
npx playwright show-trace test-results/*/trace.zip
```

Traces are recorded automatically on first retry (`trace: "on-first-retry"` in
`playwright.config.ts`).
