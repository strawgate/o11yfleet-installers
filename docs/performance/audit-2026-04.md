# Performance Audit — 2026-04

Reviewer: line-by-line scan of `packages/core`, `apps/worker`, `apps/site`,
and `packages/db`. Focus on hot paths (OpAMP heartbeat, DO event loop, API
fan-out, frontend refetch loops).

Severity model:

- **Critical** — affects every WS message or every API request; either causes
  measurable extra work per agent or breaks behavior (e.g. duplicate writes).
- **High** — visible at fleet scale (≥ 1K agents) or every page render.
- **Medium** — small constant-factor wins, but easy fixes.

A failing-test harness for several of these lives in
`packages/core/test/perf-audit.test.ts` (this branch). Each `describe` block
is `.skip`-ped by default so the standard `pnpm test` gate stays green;
remove `.skip` to reproduce a current failure or to convert the test into a
regression gate after the fix lands.

## Critical (1–20)

### 1. R2 re-upload on dedup hit — `apps/worker/src/config-store.ts:84-92`

`uploadConfigVersion` already does the R2 PUT inside the `if (!deduplicated)`
branch on lines 59-67. Lines 84-92 then run **another** `FP_CONFIGS.put` only
when `deduplicated === true`. The branches are mutually exclusive, so a dedup
hit (the expected path for repeated rollouts) incurs **1 unnecessary R2 PUT**
where 0 was intended — the bytes are already in R2 by content hash. Over
many rollouts this is wasted PUTs and added latency on every dedup'd publish.

Fix: remove lines 84-92. They have no other effect.

Failing test: `perf-audit.test.ts > "uploadConfigVersion does not re-upload on dedup hit"`.

### 2. `encodeFrame` byte explosion via `Array.from(Uint8Array)` — `packages/core/src/codec/framing.ts:18-26`

`JSON.stringify(msg, replacer)` for `Uint8Array` returns
`{ __type: "bytes", data: Array.from(value) }`. Each byte is rendered as a
decimal JSON number. A 16-byte `instance_uid` becomes `[171,171,...,171]`
(~64 chars) plus the wrapper, where base64url would be ~22 chars and raw
binary 16. Over the JSON codec on every WS heartbeat with `instance_uid` and
two 32-byte SHA hashes this **inflates the WS frame by 5–10× per message**.

Fix: encode `Uint8Array` as base64url; decode with `atob`. Same change in the
reviver.

Failing test: `perf-audit.test.ts > "encodeFrame is compact for Uint8Array fields"`.

### 3. Hand-rolled SHA-256 in JS — `packages/core/src/state-machine/processor.ts:27-114`

Cloudflare Workers ship `crypto.subtle.digest("SHA-256", ...)` (BoringSSL,
native). The 90-line JS loop is invoked from `processFrame` whenever a
collector reports an `effective_config` block (line 261), to detect a change
in the running config. With 50K agents and the default heartbeat cadence,
agents that re-report effective config trigger this routinely. For a 64KB
collector config that's ~1000 SHA-256 rounds in JS per such report.
Empirically `subtle.digest` is 10–100× faster on Workers.

Caveat: `subtle.digest` is async; `processFrame` is sync. Either move SHA out
of `processFrame` (compute once on the caller side, before invoking) or make
`processFrame` async.

### 4. `getConfigBytes` re-encodes YAML on every WS message — `apps/worker/src/durable-objects/config-do.ts:157-160`

```ts
private getConfigBytes(): Uint8Array | null {
  const config = this.getDesiredConfig();
  return config.content ? new TextEncoder().encode(config.content) : null;
}
```

Allocates a fresh `TextEncoder` and re-encodes the entire YAML on every single
heartbeat. With 50K agents × heartbeats this is hundreds of MB/s of redundant
encoding. The desired config rarely changes — cache the encoded bytes keyed by
`config.hash` on the DO instance.

### 5. `webSocketMessage` reads desired config twice — `apps/worker/src/durable-objects/config-do.ts:437-438`

```ts
const config = this.getDesiredConfig();
const configBytes = this.getConfigBytes();
```

`getConfigBytes()` calls `getDesiredConfig()` again. Two SQL `SELECT` per
heartbeat where one suffices. Keep one local and pass content through.

### 6. `bufferEvents` triggers a `COUNT(*)` on every event batch — `apps/worker/src/durable-objects/agent-state-repo.ts:570-584`

```ts
export function bufferEvents(...) {
  ...
  trimOverflowEvents(sql);   // unconditional
}
```

`trimOverflowEvents` runs `SELECT COUNT(*) FROM pending_events` synchronously
every time. On a heartbeat that emits 1 event this is a wasted full scan.
Track the row count in `do_config` and only trim when the cached count crosses
the threshold; or trim only inside `pruneEvents` (alarm tick).

### 7. `trimOverflowEvents` uses `DELETE WHERE id NOT IN (subselect …)` — `agent-state-repo.ts:586-598`

The `id NOT IN (SELECT id FROM pending_events ORDER BY id DESC LIMIT ?)`
pattern is the slowest possible delete shape in SQLite. It materializes the
keep-set, then probes every row with NOT IN. Replace with:

```sql
DELETE FROM pending_events
WHERE id <= (SELECT MAX(id) - ? FROM pending_events)
```

(or keep a stable cutoff in the do_config row).

### 8. `summarizeTextDiff` allocates 2× YAML to compute byte length — `apps/worker/src/routes/v1/index.ts:725-738`

```ts
size_bytes_delta:
  new TextEncoder().encode(latest).byteLength - new TextEncoder().encode(previous).byteLength,
```

Creates two `TextEncoder` objects and two full Uint8Array copies of YAML up
to 256KB each — just to ask for `.byteLength`. Use a UTF-8 byte counter
that does not allocate, or skip the metric.

### 9. `longestCommonSubsequenceLength` is O(N×M) on YAML lines — `routes/v1/index.ts:769-787`

For two 256KB YAML files of ~10K lines each, that's 100M cell evaluations on
the hot worker request thread (`/api/v1/configurations/:id/version-diff-latest-previous`).
The current call site fetches the diff every time the user opens the rollout
tab. Use a Myers diff with linear-space patience or just skip diff and report
"changed" + line counts.

### 10. `crypto.subtle.importKey` re-runs per claim verify — `packages/core/src/auth/claims.ts:18-26, 28-34, 36-73`

Every WS reconnect invokes `verifyClaim` → `getSigningKey(secret)` →
`crypto.subtle.importKey`. `importKey` is the slow step (raw HMAC keys must
be parsed). Cache the imported `CryptoKey` per worker isolate keyed on the
secret (or a hash of it).

### 11. `handleGetOverview` is N+1 across DOs — `routes/v1/index.ts:800-844`

For each configuration owned by the tenant the worker does
`stub.fetch("http://internal/stats")`. Every DO call internally runs 4
aggregation queries. With 50 configs that's 200 DO-SQL queries per page load.
Cache stats on D1 with a short TTL or batch via a shared aggregation DO.

### 12. `handleListTenants` materializes 2 derived tables — `routes/admin/index.ts:202-232`

```sql
LEFT JOIN (SELECT tenant_id, COUNT(*) … GROUP BY tenant_id) c ON c.tenant_id = t.id
LEFT JOIN (SELECT tenant_id, COUNT(*) … GROUP BY tenant_id) a ON a.tenant_id = t.id
```

D1 will materialize the entire `agent_summaries` and `configurations`
aggregates before joining. Even with `LIMIT 100` the inner scans are full.
Use correlated scalar subqueries or maintain counters on `tenants`.

### 13. `handleListTenants` defeats indexes with `LOWER(...)` — `routes/admin/index.ts:189-194`

```sql
LOWER(t.name) LIKE LOWER(?) ESCAPE '\'
```

Both sides wrapped in `LOWER` — even with `COLLATE NOCASE` an index on `name`
would not be hit. Use the index by storing/normalizing case once and using
`name COLLATE NOCASE LIKE ?`.

### 14. `handleUpdateTenant` does `SELECT *` 3× — `routes/admin/index.ts:249-292` and `routes/v1/index.ts:848-865`

Pattern: SELECT for ownership check, UPDATE, SELECT to return updated. With
D1's per-query overhead this is 3× the round-trips needed. Use:

```sql
UPDATE tenants SET … WHERE id = ? RETURNING *
```

### 15. `getOwnedConfig` is `SELECT *` and called per write — `routes/v1/index.ts:188-199`

Every per-config handler starts by reading the entire `configurations` row
just to check ownership. The single UPDATE/DELETE `WHERE id=? AND tenant_id=?`
already enforces ownership; check `result.meta.changes` instead.

### 16. `handleDeleteConfiguration` deletes R2 objects sequentially — `routes/v1/index.ts:343-371`

```ts
for (const { r2_key: r2Key } of versions.results) {
  await deleteConfigContentIfUnreferenced(env, r2Key);
}
```

Sequential `await` on R2 deletes: with 100 versions that's 100× the
round-trip-time. Wrap in `Promise.all`. Note that
`deleteConfigContentIfUnreferenced` itself does multiple R2/D1 round-trips
that should also be parallelized.

### 17. Front-end `useConfigurations` polls every 10s — `apps/site/src/api/hooks/portal.ts:166`

`refetchInterval: 10_000` plus `AgentsPage.tsx` rendering one
`<AgentSection>` per config (each with its own `useConfigurationAgents` and
`useConfigurationStats`) means every open portal tab fires `~3 × N + 1`
requests every 10 seconds, even when the page is idle. With 100 configs this
is ~30 req/s per tab. Use `refetchOnWindowFocus`, longer interval, or push
updates from the worker.

### 18. `AgentsPage` filters the agent array three times — `apps/site/src/pages/portal/AgentsPage.tsx:51-53`

```ts
const connectedCount = (agents ?? []).filter(...).length;
const healthyCount = (agents ?? []).filter(...).length;
const degradedCount = (agents ?? []).filter(...).length;
```

Three full scans of the array on every render. Single reduce, or get these
from `useConfigurationStats`.

### 19. `pageContext` is rebuilt every render and breaks `useMemo` — `AgentsPage.tsx:55-92,126-143`

`pageContext` is built unconditionally (no `useMemo`) and is then a
dependency of the `browserContext` `useMemo`. Reference identity changes each
render so the memo never hits. Same pattern in
`pages/portal/ConfigurationDetailPage.tsx:267-313`. Memoize `pageContext`
itself or move it inside the same memo.

### 20. `handleSetDesiredConfig` re-encodes per socket — `apps/worker/src/durable-objects/config-do.ts:633-670`

`buildConfigMap` (which `TextEncoder().encode`s the YAML) is correctly
called once outside the socket loop, so YAML encoding is not duplicated.
However, `encodeServerToAgent` is called per socket and re-runs the full
JSON-replacer (or protobuf-encoder) walk over the entire `ServerToAgent`,
including the YAML body Uint8Array. Combined with #2 (`Array.from`
expansion), the per-socket cost is `O(yaml_size)` even though only
`instance_uid` differs between sockets. Build the per-codec serialized
prefix and suffix once, then only swap the instance UID per socket. For an
N-agent rollout the today is `O(N × yaml_size)` JSON-stringify work; the
fix makes it `O(yaml_size + N × uid_size)`.

## High (21–40)

### 21. `loadAgentState` reads `SELECT *` — `agent-state-repo.ts:203`

Returns every column including the legacy `effective_config_body` column
(only kept for rollback compat per migration #4) and the `rate_window_*`
counters not used by the caller. Hot path on every WS message; switch to an
explicit column list.

### 22. Two `COUNT(*)` per alarm tick — `agent-state-repo.ts:587, 646`

`pruneEvents` and `countPendingEvents` both `SELECT COUNT(*)`. Combine into
one read or cache.

### 23. `handleGetStats` runs 4 aggregate queries — `agent-state-repo.ts:328-407, config-do.ts:690-707`

`getStats` + `getCohortBreakdown` total: COUNT/SUM agents, COUNT drifted,
GROUP BY status, GROUP BY current_config_hash. Combine into one CTE.

### 24. `tenants.created_at` not indexed — `packages/db/migrations/0001_initial.sql`

`handleListTenants` orders by `created_at DESC`. Add
`CREATE INDEX idx_tenants_created_at ON tenants(created_at DESC, id DESC)`.

### 25. ~~`tenants.plan` not indexed~~ — **downgraded**

`handleListTenants` does filter by `plan = ?`, but the column is low-
cardinality (5 plan values) and the table is small (≪ 10K rows in any
realistic deployment). Sequential scan is acceptable here; defer indexing
until a profile shows it matters. Worth keeping on the radar but not a fix
target today.

### 26. `configurations.created_at` not indexed — `0001_initial.sql`

`handleListConfigurations` orders by it. Add a composite index `(tenant_id, created_at DESC)`.

### 27. `config_versions.created_at` not indexed — `0001_initial.sql`

`handleListVersions`, `handleLatestVersionDiff` order by `created_at DESC, rowid DESC`.
Add `(config_id, created_at DESC)` index.

### 28. Redundant index on `enrollment_tokens.token_hash` — `0001_initial.sql`

Column is `UNIQUE` — SQLite already creates an index. The explicit
`idx_enrollment_tokens_hash` is duplicate storage and write overhead.

### 29. `hexToUint8Array` slow per-byte — `packages/core/src/hex.ts:3-12`

Uses `parseInt`+`String.substring` per byte. With `instance_uid` (16 bytes)
× every WS message this is hot. Use a 256-entry lookup table.

### 30. `uint8ToHex` slow per-byte — `hex.ts:14-18`

Allocates `Array.from(arr)`, maps to padded hex string, joins. Same hot path.
Use a 256-entry lookup table; or reuse a single buffer.

### 31. `TextEncoder/Decoder` re-allocated in `processor.ts` — `processor.ts:43, 259`

Inside `sha256Hex` and the effective-config decode, both run per heartbeat.
Hoist to module scope.

### 32. `JSON.stringify(agent_description)` per heartbeat — `processor.ts:246-251`

Computed unconditionally to compare against `state.agent_description`.
For an agent_description with 50 attributes this is non-trivial. Compare a
short hash, or compare structurally and only stringify on change.

### 33. `Object.values(configMap)[0]` allocates — `processor.ts:257`

Allocates an array of all values just to take index 0. Use a `for…in` and
`break` after first.

### 34. `getStats` SUM-CASE aggregates over the `agents` table — `agent-state-repo.ts:328-348`

Three `SUM(CASE WHEN …)` aggregates over the entire `agents` table on every
dashboard refresh. With ≤ 50K rows per DO and DO-local SQLite (~µs/row) this
is real CPU but not a hot-path emergency. Severity is Medium-High, not
Critical — fix after #23 (combine `getStats` + `getCohortBreakdown`) lands,
since that consolidation will absorb most of this work.

### 35. `listAgentsPage` `q LIKE %term%` — `agent-state-repo.ts:466-500`

LIKE with leading wildcard cannot use any index. Adopt FTS5 virtual table
or restrict to prefix match.

### 36. `pruneEvents` lacks `created_at` index — `agent-state-repo.ts:604-613`

`DELETE FROM pending_events WHERE created_at < ?` does a full scan. Add
`CREATE INDEX idx_pending_events_created_at ON pending_events(created_at)`.

### 37. `handleUpdateConfiguration` is 3 round-trips — `routes/v1/index.ts:312-341`

SELECT (ownership), UPDATE, SELECT (return). Use `RETURNING *`.

### 38. `handleImpersonateTenant` SELECT/INSERT/SELECT for the impersonation user — `routes/admin/index.ts:710-787`

When the impersonation user does not exist, the handler runs SELECT (line
715), then `INSERT OR IGNORE` (737), then SELECT again (749). Use
`RETURNING *` on the INSERT to drop the second SELECT. The first SELECT is
still needed to short-circuit when the user already exists.

### 39. `handleSeed` re-hashes passwords on every run — `routes/auth.ts:220-289`

Even when the seeded password hasn't changed, runs PBKDF2 100k twice (tenant

- admin). Skip when `verifyPassword(seedPassword, existing.password_hash)`
  matches.

### 40. `generateSessionId` builds hex via Array.from — `routes/auth.ts:48-53` and `routes/admin/index.ts:29-34`

Same anti-pattern as `uint8ToHex`. Per login, per impersonation. Reuse a
helper backed by a precomputed table.

## Medium (41–50)

### 41. `timingSafeEqual` allocates `TextEncoder` per call — `apps/worker/src/utils/crypto.ts:3-7`

Every Bearer-token request hits this. Hoist the encoder.

### 42. `scheduled` cron loads all configs unbounded — `apps/worker/src/index.ts:222-256`

`SELECT id, tenant_id FROM configurations` with no `LIMIT`. For 100K configs
this is one massive D1 round-trip plus 100K DO fan-out. Page in batches.

### 43. `mapWithConcurrency` duplicates the input array — `apps/worker/src/index.ts:131-150`

`indexedItems = items.map((item, index) => ({ item, index }))` allocates a
parallel array. Use a shared cursor and read `items[i]` directly.

### 44. `handleHealthCheck` is 11 D1 queries per probe — `routes/admin/index.ts:435-540`

Even batched, this is steady D1 cost from monitors. Cache for 30s or move to
a materialized counters table.

### 45. `handleListPlans` GROUP BY tenants on every load — `routes/admin/index.ts:651-670`

Tenant counts by plan recomputed each call. Cache or back with a counter.

### 46. `decodeFrame` JSON.parse runs reviver per value — `packages/core/src/codec/framing.ts:54-62`

The reviver fires for every key, even non-bytes. With deeply nested
agent_description this adds up. Walk the parsed object once after parsing.

### 47. ~~`base64urlEncode` builds string by `+=` concat~~ — **withdrawn**

Verification: the function already uses `String.fromCharCode(data[i])` per
byte and `btoa(binary)`. The `+=` over single-char strings on a string up to
~44 chars long (32-byte HMAC) is fine and the overall pattern is the
recommended idiom for Workers. No action.

### 48. Extra Uint8Array copy in `encodeServerToAgentProto` — `packages/core/src/codec/protobuf.ts:158-167`

Allocates `result = new Uint8Array(1 + payload.length)`, copies payload, then
`result.buffer.slice(...)` — a second copy. Replace with `Uint8Array.from`
construction or hand the existing buffer through.

### 49. `handleGetOverview` masks per-DO failures — `routes/v1/index.ts:805-823`

The per-DO `Promise.all` task wraps the `stub.fetch` in a try/catch that
returns `{ total: 0, connected: 0, healthy: 0 }` on any error, so failures
look identical to "no agents" downstream. This is also a perf/availability
risk: with no per-call timeout, a slow DO can hold up the whole batch (it
won't reject, but the await blocks the parent). Add a timeout per DO call,
distinguish failure from empty stats in the response, and consider
`Promise.allSettled` to surface per-config errors.

### 50. `<AgentSection>` not virtualized — `apps/site/src/pages/portal/AgentsPage.tsx:311`

`cfgList.map((c) => <AgentSection ... />)` renders one fully-mounted section
per configuration. Each section keeps its own React-Query cache, runs its own
`useEffect`s, and re-renders on every refetch. With many configs this dominates
the render budget. Lazy-mount visible sections (Intersection Observer) or
collapse to a list.

## Notes / non-issues considered

- DO `_attachment_` reuse via `serializeAttachment` is fine — the attachment
  is small.
- `processFrame` clones state with `{ ...state, ... }`. Allocations are
  consistent and not a hot bottleneck after the higher-priority fixes.
- The WebSocket auto-response `ping/pong` setup at `config-do.ts:142` is the
  right primitive — it keeps idle WS alive without waking the DO, exactly
  matching the design boundary in `docs/architecture/overview.md` (Queue/DO
  hot path stays free of D1 writes).

## Verification pass (2026-04-30)

Each finding was re-checked against the current source by independent
sub-agents. Adjustments applied above:

- **#1 R2 dedup**: Corrected from "2 PUTs" to "1 unnecessary PUT (vs
  expected 0)" — the branches are mutually exclusive (CodeRabbit catch).
- **#3 SHA-256 frequency**: Reframed — runs whenever a collector reports
  `effective_config`, not on every heartbeat unconditionally.
- **#20 per-socket re-encode**: Clarified — `buildConfigMap` (the YAML
  encode) is correctly hoisted; the redundant work is the per-socket
  JSON-replacer walk over the full message including the YAML body.
- **#25 `tenants.plan` index**: Downgraded — low cardinality + small table.
- **#34 SUM-CASE**: Reframed as Medium-High; absorbed by the #23 CTE
  consolidation.
- **#38 impersonation**: Clarified — SELECT/INSERT/SELECT rather than
  "twice in a row".
- **#47 base64urlEncode**: Withdrawn — current code already uses the
  recommended `String.fromCharCode` + `btoa` pattern.
