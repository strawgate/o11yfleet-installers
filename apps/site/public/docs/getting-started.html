Getting started

Get up and running with O11yFleet

Start the control plane, create a collector configuration, connect one OpenTelemetry Collector, and verify that O11yFleet can see it.

Quick option: If you just want to try O11yFleet locally, copy apps/worker/.dev.vars.example to apps/worker/.dev.vars and run just dev-up. It starts the local Worker and site, migrates local D1, seeds a demo tenant, uploads a sample config, and creates an enrollment token. No hosted account needed.

Step 1 — Set up O11yFleet

Choose either hosted access if your workspace has already been provisioned, or run locally for development/testing. Self-service registration is not enabled yet.

Local development:

git clone https://github.com/strawgate/o11yfleet.git
cd o11yfleet
just install
cp apps/worker/.dev.vars.example apps/worker/.dev.vars
just dev-up

This starts the Worker at http://localhost:8787, the site at http://127.0.0.1:3000, applies local migrations, and seeds demo data. The default local tenant login is demo@o11yfleet.com / demo-password; the admin login is admin@o11yfleet.com / admin-password.

Step 2 — Create an enrollment token

Enrollment tokens bootstrap collector authentication with O11yFleet. After the initial connection, the collector receives a signed claim that lasts for 24 hours and can be used for reconnects.

In the portal, open Enrollment tokens and create a token for your configuration. For local or scripted setup, use the CLI: 

# If you ran `just setup`, a demo tenant already exists
# Create an enrollment token for a configuration
ofleet token:create --config-id <config-id> --label "prod-ingress" --expires-in 24

Copy the token while it is visible. Tokens are stored hashed, so O11yFleet cannot show the raw value later.

Step 3 — Connect a collector

Now you'll run an OpenTelemetry Collector with OpAMP enabled and point it at O11yFleet.

  1. Install the OpenTelemetry Collector binary (if you haven't already): 
    # macOS (Homebrew)
brew install open-telemetry/opentelemetry-collector/opentelemetry-collector

# Linux (binary)
curl -L https://github.com/open-telemetry/opentelemetry-collector-releases/releases/download/v0.105.0/otelcol-contrib_0.105.0_linux_amd64.tar.gz | tar xz

# Docker
docker run -p 4317:4317 otel/opentelemetry-collector-contrib:latest
  2. Create a config file that enables OpAMP: 
    receivers:
  otlp:
    protocols:
      grpc:
      http:

exporters:
  otlp:
    endpoint: "your-backend:4317"

service:
  telemetry:
    logs:
      level: "info"
  extensions:
    - opamp
  pipelines:
    traces:
      receivers: [otlp]
      exporters: [otlp]

extensions:
  opamp:
    server:
      ws:
        endpoint: "ws://localhost:8787/v1/opamp"   # or wss://api.o11yfleet.com/v1/opamp for SaaS
    headers:
      Authorization: "Bearer YOUR_TOKEN_HERE"
  3. Run the collector: 
    ./otelcol-contrib --config=config.yaml

Within a few seconds, the collector should appear in your O11yFleet portal with a green healthy status.

Step 4 — Verify connection

In the O11yFleet portal, go to Agents. You should see your newly connected collector listed with:

  • Status: Connected
  • Health: Healthy (green dot)
  • Version: The collector version you're running
  • Heartbeat: "just now" or "a few seconds ago"

If the collector shows as offline or degraded, check:

  • WebSocket connection in the collector logs
  • Enrollment token validity (expired or revoked tokens are rejected)
  • Network/firewall rules allowing outbound WebSocket to api.o11yfleet.com:443 (SaaS) or localhost:8787 (local)
  • Correct OpAMP extension configuration

Step 5 — Manage your collector

Now that your collector is connected, you can:

  • View health — connection status, last seen, version
  • See effective config — what configuration the collector is actually running
  • Edit the config — upload a new YAML file, then start a rollout
  • Roll out updates — push the current config to all connected agents immediately (progressive delivery coming soon)
