Guide

GitOps workflow

Connect a GitHub repository to a configuration. O11yFleet watches the repo and automatically creates new versions when config files change — giving you Git-powered audit trails, PR reviews, and rollback.

Connect a repository

  1. Open a configuration in the O11yFleet portal
  2. Go to Settings → Git
  3. Click Connect GitHub repository
  4. Authorize O11yFleet (OAuth scopes: read repository contents, read webhook payloads)
  5. Select the repository and branch to watch
  6. Enter the path to the configuration file (e.g., otel/prod-ingress.yaml)
  7. Save
Multi-repo, per-config: Each configuration can be backed by a different repo or branch. Production configs can require PRs, while sandbox configs allow direct commits.

How it works

When you connect a repository, O11yFleet registers a webhook with GitHub. On every push to the configured branch, O11yFleet:

  1. Fetches the config file from the specified path
  2. Validates YAML syntax and schema
  3. Compares against the current version to detect changes
  4. Creates a pending version in O11yFleet
  5. Lets you review changes and approve the rollout

Commit signing & required reviews

For production configurations, you can enforce:

  • Signed commits — commits must be GPG-signed
  • Required reviewers — at least N approved reviews before merge
  • Status checks — CI must pass
  • CODEOWNERS — specific teams must approve

These are enforced by GitHub branch protection rules; O11yFleet respects them and won't auto-deploy if the commit doesn't meet policy.

Approval flow

  1. Push to Git — A developer merges a PR with config changes
  2. Webhook fires — GitHub notifies O11yFleet
  3. Version created — New pending version appears in portal
  4. Review — Team reviews the YAML diff and health impact
  5. Rollout — Click Roll out to begin an immediate global rollout; progressive delivery is planned

Manual override

You can always edit a configuration directly in the portal (UI edits take precedence). If you do, the Git-backed mode remains connected — the next Git push will create a new pending version, which you can choose to deploy or discard.

This lets you handle emergencies in the UI while maintaining Git as the long-term source of truth.
