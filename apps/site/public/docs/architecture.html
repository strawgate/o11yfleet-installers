Architecture

O11yFleet is built on Cloudflare's edge platform using Workers, Durable Objects, D1, R2, and Queues. This architecture provides global distribution, automatic scaling, and sub-millisecond latency for agent management.

System overview

The system has three main planes:

  1. Agent Control Plane — Durable Objects manage real-time agent connections, config delivery, health tracking
  2. Management API — Worker-based REST API for tenant/config/token CRUD
  3. Auth Layer — Session-based auth for portal, Bearer token for programmatic access

Key design decisions

Config-as-Durable-Object partitioning

Each (tenant_id, config_id) pair maps to one Durable Object. This gives us:

  • Natural sharding — each config group gets its own DO
  • Zero-cost hibernation — idle DOs sleep; no CPU charges
  • Per-config state — DO internal SQLite stores agent state without hitting D1
  • Geographic distribution — Cloudflare automatically runs DOs close to agents

Signed assignment claims

After enrollment, each agent receives a signed HMAC-SHA256 claim containing tenant_id, config_id, instance_uid, and generation. Hot-path connections verify locally — no D1 lookup. Only enrollment hits the database.

R2 content-addressed config storage

Config YAML is stored in R2 keyed by SHA-256 hash:

  • Key format: configs/sha256/{hash}.yaml
  • Identical configs share one R2 object (deduplication)
  • Config DO references hash; never stores full YAML

Queue-based analytics

Config DO emits events to a Queue on agent state changes. The current consumer writes analytics datapoints to Analytics Engine. Agent state for API reads lives in DO-local SQLite; the old D1 agent_summaries read model is not used.

Infrastructure

The entire stack is defined as Infrastructure as Code using Terraform. Cloudflare resources (Workers, Durable Objects, D1 databases, R2 buckets, Queues) are provisioned automatically on deploy.

See infra/terraform/ in the repository for full definitions.
